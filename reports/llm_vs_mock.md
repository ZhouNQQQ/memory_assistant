# LLM vs Mock 对比报告

> 生成日期: 2026-06-13
> 评估样本: 5 条真实对话（来自 1.md，SKILL 文档优化场景）
> 模型: GLM-4-Flash
> 评估方法: Precision / Recall / F1，基于人工标注 ground truth

---

## 一、总体指标

| 方法 | Precision | Recall | F1 |
|------|-----------|--------|-----|
| **Mock（关键词匹配）** | 0.11 | 0.08 | **0.10** |
| **LLM（GLM-4-Flash）** | 0.60 | 0.75 | **0.67** |
| **提升** | +0.49 | +0.67 | **+0.57 (570%)** |

> **Token 消耗**: 5 轮对话共消耗 5,727 tokens（prompt 4,779 + completion 1,028）

---

## 二、Case 级对比（≥4 例）

### Case 1：Dialog 0 — 用户列出 5 个上下文优化问题

**Ground Truth**（标注 3 条）：
1. 用户的目的之一是减少 AI 助手上下文的使用
2. 用户发现 SKILL-02 Step2 要求"先取签名定位"，但自己没有做到
3. 用户经常 read_file 读取整个文件而非只取签名

**Mock 提取**（6 条，全错）：
- "打算求"、"打算写进方案的那个方法精读实现"... — 全是"打算"关键词误触发，提取的是文本中的零散片段，不是用户陈述的事实

**LLM 提取**（6 条，命中 2 条）：
- ✅ 命中 GT-2："用户在处理 PaxJourneyDetailServiceImpl 等文件时，未遵循先取签名定位..."
- ✅ 命中 GT-3："用户在多个 feature 中重复读取了同一文件..."
- ❌ 漏了 GT-1："减少上下文使用"这个目的性陈述
- ❌ 多提取了 2 条：grep 截断浪费、子 Agent 回灌较长（也是事实，但用户没标注）

**Mock 错因**：关键词"打算"在用户文本中多次出现，但都是 SKILL 文档中的示例文本，不是用户自己的陈述。Mock 分不清"用户说的话"和"用户引用的文档内容"。

---

### Case 2：Dialog 1 — 用户决定优化第 3、5 条

**Ground Truth**（2 条）：
1. 用户决定优先优化第 3 条（grep 排除测试）和第 5 条（回灌收敛量化）
2. 用户决定暂不处理跨 feature 复用，认为影响主会话上下文不大

**Mock 提取**：0 条（无匹配关键词）

**LLM 提取**（2 条，全中）：
- ✅ "用户想要优化第3条和第5条"
- ✅ "用户决定先不做'2 跨feature复用'，因为它主要在子agent中使用..."

**Mock 错因**：无"名字/偏好/职业/计划/关系"关键词，直接 0 输出。规则引擎在结构化决策文本上完全失效。

---

### Case 3：Dialog 4 — 用户问能否并行设计 feature

**Ground Truth**（2 条）：
1. 用户认为处理多个 feature 时执行速度较慢
2. 用户考虑使用子 Agent 并行设计多个 feature 来提速

**Mock 提取**：0 条

**LLM 提取**（2 条，全中）：
- ✅ "用户认为多个feature的需求执行较慢"
- ✅ "用户询问是否可以使用子agent并行设计这多个feature"

**Mock 错因**：同上，无关键词匹配。

---

### Case 4：Dialog 5 — 用户关心子 Agent 确认问题怎么处理

**Ground Truth**（2 条）：
1. 用户认可子 Agent 并行设计的建议
2. 用户关心子 Agent 产生的用户确认问题如何处理流程

**Mock 提取**：0 条

**LLM 提取**（2 条，命中 1 条）：
- ✅ "用户对子agent中产生的需要用户确认的问题的处理方式有疑问"
- ❌ 多提取了"用户输入后的执行方式有疑问"（也是事实，但用户没标注）

**Mock 错因**："认可""关心"这类情感/态度词不在关键词列表中。

## 三、设计缺陷发现

### 缺陷 1：评估函数的语义匹配 vs 字符串匹配（本项目特有）

**发现过程**：第一次运行评估时，LLM F1 = 0.00（全部不匹配）。检查后发现：
- LLM 输出："用户认为多个feature的需求执行较慢"
- Ground Truth："用户认为处理多个 feature 时执行速度较慢"
- 两者语义几乎相同，但字符串不完全匹配

**问题**：用简单的字符串标准化（去掉停用词）做匹配，在中文场景下仍然无法处理同义改写。

**临时修复**：放宽匹配函数，增加公共子串检查（≥4 字）。

**更优方案**：用 Embedding 相似度做匹配（如 sentence-transformers），但会增加评估复杂度。当前场景下，人工标注的 ground truth 本身带有主观性，F1 仅供参考。

---

### 缺陷 2：Mock 的关键词规则完全无法覆盖技术/工作场景

**发现过程**：Mock 的规则基于"名字、偏好、职业、计划、关系"，但用户的对话全是关于 SKILL 文档优化、子 Agent 设计、交互模板改进等技术工作场景。Mock 的 5 类规则全部失效，F1 只有 0.10。

**启示**：规则引擎的价值取决于规则的领域覆盖度。在通用对话场景下，规则永远覆盖不全。这也是为什么 LLM 提取（F1=0.67）远超规则（F1=0.10）的核心原因。

---

### 缺陷 3：LLM 提取存在"多提取"和"漏提取"

**多提取**：LLM 在 Dialog 0 提取了 6 条（用户文本中确实有 5 个优化点），但 ground truth 只标注了 3 条。这导致 Precision 不是 1.0。

**漏提取**：LLM 在 Dialog 0 漏了"用户目的是减少上下文使用"这个高层目标，只提取了具体的 5 个优化点。说明 LLM 倾向于提取"具体事实"而非"抽象目标"。

---

## 四、5 分钟面试口述要点

> 背下面这段话，面试时 5 分钟讲完：

"我用真实 LLM（GLM-4-Flash）和关键词 Mock 提取做了对比评估。样本是 5 条真实工作对话，人工标注了 ground truth。

Mock 的 F1 只有 0.10——因为规则引擎基于名字、偏好、职业、计划、关系，完全无法覆盖技术工作场景中的决策、问题、优化点。LLM 的 F1 是 0.67，提升了 570%。

但评估过程中我发现一个关键缺陷：同义改写的语义匹配，用字符串标准化无法处理。比如 LLM 提取'用户认为多个feature的需求执行较慢'，ground truth 是'用户认为处理多个 feature 时执行速度较慢'，语义相同但字符串不同。这说明**评估函数本身也需要语义能力**，不是简单的字符串匹配。

另外，LLM 倾向于提取具体事实（'你违反了读码归属唯一性'），但会漏掉抽象目标（'你的目的是减少上下文使用'）。这个偏差需要在 prompt 里明确告诉 LLM 也要提取'高层意图'。"
