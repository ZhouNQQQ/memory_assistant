## 批改

### 🟢 正确的点

1. **核心库不做 Compaction 是设计惯例** —— 正确。类比：Chroma 向量库也不做，Redis 也不做。基础设施层只提供基础操作，策略层让用户自己实现。
2. **Background Review 触发** —— 正确。每 N 轮对话回顾后触发 compaction。
3. **容量阈值触发** —— 正确。"对话长度超过 X 条"是容量阈值，超出后淘汰低价值记忆。
4. **幂等恢复** —— 正确。打标 + 跳过，避免重复处理。

### 🟡 可优化的点

| 你说的 | 优化建议 |
|--------|----------|
| "行业管理" | 应为"行业惯例"或"设计惯例" |
| 缺了"定时触发" | 完整答案应包含三种：①定时（每天/每周）、②容量阈值（超 X 条）、③Background Review（每 N 轮） |
| 幂等恢复缺了补充 | 误删了怎么办？从 history 表恢复（mem0 update 保留旧版本到 history） |

### 完整答案（30 秒版）

> "Compaction 有三种触发时机：①定时任务（如每天凌晨跑清理）；②容量阈值（记忆超 1 万条，淘汰 bottom 10%）；③Background Review（每 N 轮对话后触发，回顾+压缩+清理）。
> 失败恢复用幂等设计：给每条记忆打 `compacted_at` 时间戳，已处理的跳过，只处理未标记的。误删了从 history 表恢复。"

## 关键知识

### 三种触发时机对比

| 触发方式 | 频率 | 职责 | 智慧程度 |
|----------|------|------|----------|
| **定时** | 每天/每周 | 清理过期记忆、合并重复 | 无（纯规则） |
| **容量阈值** | 超 X 条触发 | 淘汰低价值记忆 | 无（按分数排序） |
| **Background Review** | 每 N 轮对话 | 回顾后判断哪些过时、哪些可合并、哪些可升级为技能 | 有（LLM 参与） |

### 幂等实现

```python
# 伪代码
for mem in candidates:
    if mem.get("compacted_at"):
        continue  # 已处理，跳过
    
    do_compact(mem)
    mem["compacted_at"] = now()
    save(mem)

# 失败重跑时：已标记的跳过，未标记的继续处理
```
