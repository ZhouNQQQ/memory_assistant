## 一、Rolling Summary（热/冷路径分层摘要）

### 1. 为什么需要分层？

对话记录会无限增长，全部塞进上下文 = 撑爆 token。但全部压缩成一条摘要 = 丢失细节。

**分层思路**：
- 最近的是精确的（热路径）
- 远的是摘要的（冷路径）
- 查询时按需组合

### 2. 热路径（Hot Path）

**定义**：最近 N 轮（如 10-20 轮）的精确对话记录

**特点**：
- 保留原始文本，不压缩
- 查询时优先检索
- 确保最新信息不丢失（如"我刚刚说的"）

**分界标准**：
- 按轮数：最近 10-20 轮
- 按时间：最近 1 小时内的对话
- mem0 的做法：按会话（session）区分，当前会话的消息是热路径

### 3. 冷路径（Cold Path）

**定义**：超过 N 轮的历史对话，分层压缩存储

**分层结构**：

```
原始对话（轮级）
    ↓ 每天压缩一次
日摘要（保留关键事件 + 实体）
    ↓ 每 7 天合并一次
周摘要（保留重要决策 + 关系变化）
    ↓ 每 30 天合并一次
月摘要（保留长期趋势 + 用户画像）
```

**各层压缩策略**：

| 层级 | 保留什么 | 丢弃什么 | 压缩率 |
|------|----------|----------|--------|
| 日摘要 | 关键事件、具体决策、新实体 | 寒暄、重复确认 | 10:1 |
| 周摘要 | 重要决策、关系变化、偏好变化 | 日常琐事、已执行的计划 | 5:1 |
| 月摘要 | 长期趋势、稳定偏好、用户画像 | 临时计划、已完成的短期目标 | 3:1 |

### 4. 分层合并流程

```
每日触发（定时或对话量触发）
    ↓
取最近 24 小时的对话
    ↓
LLM 摘要："用户在 6/12 做了以下重要事情：1. 决定优化 SKILL 文档 2. 引入并行设计模式"
    ↓
检查是否与已有日摘要冲突（重复事件不合并）
    ↓
写入日摘要存储

每周触发（每 7 天）
    ↓
取最近 7 天的日摘要
    ↓
LLM 合并："本周用户主要关注 SKILL 文档优化，完成了 5 项改进，引入并行设计"
    ↓
检查关键实体是否丢失（小雨/李明/项目名等）
    ↓
写入周摘要

每月触发（每 30 天）
    ↓
取最近 4 周的周摘要
    ↓
LLM 合并："用户长期偏好：关注上下文效率、子 Agent 设计、技术流程优化"
    ↓
更新用户画像（USER.md）
    ↓
写入月摘要
```

### 5. 关键问题：全局摘要过期/漂移

**问题**：摘要合并时，LLM 可能遗漏关键实体

**示例**：
- 原始对话："我女朋友叫小雨，她喜欢瑜伽"
- 日摘要："用户有女朋友，她喜欢瑜伽"
- 周摘要："用户有女朋友"
- 月摘要："用户有稳定伴侣" ← 丢失了"小雨"这个名字！

**解决方案**：
1. **实体白名单**：重要实体（人名、地名、项目名）必须保留，不能压缩掉
2. **摘要校验**：每次合并后，检查原始对话中的关键实体是否还在摘要中
3. **分层存储**：即使压缩了高层摘要，底层日摘要仍然保留原始信息，查询时可以穿透到下层

## 二、Background Review（定期回顾机制）

### 1. 对话结束触发写入（即时型）

**触发时机**：每次对话结束

**流程**：
```
对话结束
    ↓
extractor.py 提取新记忆
    ↓
updater.py 决策 ADD/UPDATE/NOOP
    ↓
写入向量库 / 文件
```

**特点**：
- 即时：用户刚说完就处理
- 增量：只处理本次对话的新信息
- 局部：不关心历史记忆的状态

**问题**：
- 不处理历史记忆的衰减/压缩
- 不检查记忆是否重复/冗余
- 不更新全局用户画像

### 2. Background Review（定期型）

**触发时机**：
- 每 N 轮对话（如每 50 轮）
- 或定时触发（如每天凌晨）
- 或事件触发（记忆量超过阈值）

**流程（Hermes Agent 模式）**：
```
触发条件满足
    ↓
【阶段1】回顾近期记忆
    ├── 读取最近 N 轮的所有新记忆
    ├── 检查是否有可提取的技能模式（如"用户总是要求优化交互模板"→ 提取为通用技能）
    └── 检查是否有冗余记忆（同一事实被多次记录）

【阶段2】压缩冗余记忆
    ├── 合并相似记忆（3 条关于"上下文优化"的记忆 → 1 条综合）
    ├── 删除低重要性旧记忆（超过 30 天、未访问、低 importance）
    └── 更新时间衰减权重

【阶段3】更新用户画像
    ├── 从周/月摘要中提取长期偏好
    ├── 更新 USER.md（用户画像文件）
    └── 检查画像是否漂移（如"用户之前喜欢瑜伽，现在说喜欢健身"→ 更新偏好）

【阶段4】全局一致性检查
    ├── 检查是否有矛盾记忆（"用户喜欢咖啡" vs "用户讨厌咖啡"）
    ├── 标记冲突项，待用户确认
    └── 更新全局摘要
```

**特点**：
- 定期：不是每次对话都做，减少开销
- 全局：处理历史记忆的状态，不只是增量
- 治理：防止记忆膨胀、保持画像准确

### 3. 两者的对比

| 维度 | 对话结束触发写入 | Background Review |
|------|-----------------|-------------------|
| 触发时机 | 每次对话结束（即时） | 每 N 轮/定时（定期） |
| 处理范围 | 本次对话的增量 | 近期所有记忆的全局 |
| 主要任务 | 提取 + 写入新记忆 | 压缩 + 合并 + 画像更新 |
| 计算成本 | 中等（单次 LLM 提取） | 高（需要回顾大量记忆） |
| 用户体验 | 即时感知（记忆马上生效） | 无感知（后台运行） |
| 防止膨胀 | 不处理 | 主动压缩/删除 |

### 4. 为什么两者都需要？

- **只有即时写入**：记忆无限膨胀，画像不更新，查询质量下降
- **只有定期回顾**：新记忆延迟生效，用户说"我喜欢咖啡"后，下次对话可能还没写入
- **两者结合**：即时写入保证新记忆快速生效，定期回顾保证历史记忆健康

## 三、关键概念对比

| 概念 | Rolling Summary | Background Review |
|------|-----------------|-------------------|
| 目的 | 解决上下文过长 | 解决记忆膨胀/画像漂移 |
| 手段 | 分层压缩（热/冷） | 定期合并/删除/更新 |
| 触发 | 查询时按需组合 | 定时/定量触发 |
| 产出 | 查询时的上下文 | 压缩后的记忆库 + 更新后的画像 |

## 四、面试追问要点

**Q16：Rolling Summary 的热路径和冷路径怎么分界？**

> 按轮数或时间。mem0 的做法是按会话区分：当前会话的消息是热路径（精确保留），历史会话的消息进入冷路径（分层摘要）。更工业的做法是设一个阈值（如最近 10 轮或 1 小时），超过的自动压缩。

**Q17：全局摘要过期/漂移了怎么办？**

> 三招：① 实体白名单——重要实体不允许压缩掉；② 分层存储——高层摘要丢失细节时，可以穿透到下层日摘要查原始；③ 定期校验——Background Review 时检查画像和摘要是否矛盾，矛盾项标记待确认。

**Q19（新增）：Background Review 和对话结束触发写入有什么区别？**

> 见上方对比表。核心区别：即时 vs 定期，增量 vs 全局，写入 vs 治理。

**Q41（新增）：记忆系统长期运行后怎么防止膨胀？除了 Compaction 还有什么机制？**

> Background Review。Compaction 是被动触发（空间满了），Background Review 是主动治理（定期清理）。另外访问频率 + 时间衰减也是机制：旧记忆如果没有被 search 检索过，权重降低，更容易被清理。
