# CQRS + 双水位一致性（记忆系统视角）

> 口述稿 + 架构图
> 日期：2026-06-13
> 关联：mem0 `add()` / `search()` 源码 + Java 分布式经验迁移

---

## 一句话总结

记忆系统天然适合 CQRS，因为**命令端（add）低频重计算、查询端（search）高频轻延迟**，解耦后互不干扰，双水位保证查询快、最终一致。

---

## 口述稿（3 分钟版）

"记忆系统为什么天然适合 CQRS？我把它和之前做的 Java 分布式项目对比：

**第一，行为天然分离**。mem0 的 `add()` 是命令端，负责对话结束后的记忆提取、去重、写入；`search()` 是查询端，负责会话启动时的记忆检索。一个写、一个读，职责天然不同。

**第二，频率差异大**。命令端只在对话结束触发一次（写低频），查询端每轮对话都可能触发（读高频）。如果写和读耦合，写的重计算会阻塞读的轻查询。

**第三，成本差异大**。命令端要跑 LLM 提取 + embedding + 去重决策（秒级），查询端只需向量检索 top-k（毫秒级）。解耦后，查询端不会被写入拖慢。

**双水位设计**：
- 低水位（查询视图）：读向量库缓存，<<50ms，但可能读到旧数据
- 高水位（完整态）：LLM 提取完成、写入、索引刷新，几秒延迟
- Compensator：低水位没读到时，用内容 hash 做幂等键，去高水位补查，避免重复写入

和我之前做的顺风车订单系统一样：订单状态变更（命令）和订单查询（查询）分离，变更走 MQ 异步，查询读缓存，不一致时补偿。"

## 架构图（文字版）

```
用户对话结束
    ↓
【命令端】add()
    ├── Phase 0: 获取上下文（读最近 10 条消息）
    ├── Phase 1: 检索现有记忆（向量搜索 top-10，去重用）
    ├── Phase 2: LLM 提取新记忆（计算密集，秒级）
    ├── Phase 3: 决策 ADD/UPDATE/NOOP
    └── Phase 4: 写入向量库 + 刷新索引
         ↓（异步，延迟几秒）
【高水位】最终一致完整态
         ↓
【低水位】向量库缓存（快照，<<50ms）
    ↑
【查询端】search() ← 用户提问时触发
    ↓
【Compensator】如果低水位没读到最新记忆
    ├── 幂等键 = 内容 hash 或 会话 ID + 消息序号
    ├── 去高水位查"未完成的记忆队列"
    └── 返回结果 + 触发低水位刷新
```

## 面试追问要点

| 追问 | 答案 |
|------|------|
| 低水位读到旧数据怎么办？ | Compensator 去高水位补查，或返回"记忆正在更新" |
| Compensator 怎么保证幂等？ | 幂等键 = 内容 hash，相同内容不去重写入 |
| 为什么不用强一致？ | 记忆系统用户容忍"稍后才记得"，不像金融系统要求实时一致 |
| 和数据库读写分离的区别？ | 本质相同，但命令端计算更重（LLM 提取），不只是写主库 |

---

## 关键数字

- 命令端延迟：秒级（LLM 提取）
- 查询端延迟：毫秒级（向量检索）
- 双水位一致性窗口：几秒~几十秒（取决于 LLM 提取速度）
