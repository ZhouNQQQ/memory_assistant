## 一、向量记忆（Vector Memory）的局限

### 向量记忆怎么工作？

```
对话 → 提取事实 → embedding → 存入向量库
查询 → embedding → 向量相似度检索 → 返回 top-k
```

**核心假设**：语义相似 = 相关信息

### 局限 1：无法处理多跳关系

**示例**：
- 记忆1："张三是李四的领导"
- 记忆2："李四负责 X 项目"
- 查询："X 项目谁拍板？"

**向量检索**：
- "X 项目谁拍板" 和 "李四负责 X 项目" 语义相似 → 返回记忆2
- 但记忆2没有"拍板"信息，答案需要记忆1（张三是领导）+ 记忆2（李四负责）的**关系推理**
- 向量检索只返回单条记忆，无法跨记忆推理

### 局限 2：关系方向丢失

- 向量把"张三爱李四"和"李四爱张三"编码成几乎相同的向量
- 无法区分"谁爱谁"的方向

### 局限 3：结构化查询困难

- "张三的所有下属" → 向量检索只能找语义相似，不能找"有领导关系且指向张三"的实体

## 二、Graph Memory（图记忆）

### 图记忆怎么工作？

```
对话 → 提取实体 + 关系 → 存入图数据库（节点=实体，边=关系）
查询 → 图遍历（Cypher/GQL）→ 多跳推理 → 返回路径
```

**存储结构**：
- 节点（Node）：实体（人、公司、项目、地点）
- 边（Edge）：关系（领导、负责、属于、位于）
- 属性（Property）：节点/边的附加信息

### 示例：多跳关系查询

**存储**：
```
(张三)-[领导]->(李四)
(李四)-[负责]->(X项目)
(张三)-[属于]->(技术部)
```

**查询"X 项目谁拍板？"**：
```cypher
MATCH (person)-[负责]->(X项目)
OPTIONAL MATCH (boss)-[领导]->(person)
RETURN COALESCE(boss, person) as 拍板人
```

**结果**：张三（通过多跳推理：张三领导李四，李四负责 X 项目）

## 三、向量记忆 vs 图记忆对比

| 维度 | 向量记忆 | 图记忆 |
|------|----------|--------|
| **存储** | 连续向量空间 | 离散节点+边 |
| **查询方式** | 语义相似度检索 | 结构化遍历 |
| **擅长** | 语义匹配、模糊查询 | 关系推理、多跳查询 |
| **不擅长** | 关系推理、方向性 | 语义相似度、模糊匹配 |
| **典型场景** | "我喜欢什么" | "X 项目谁拍板" |
| **存储成本** | 中等（embedding + 向量库） | 高（图数据库 + 关系维护） |
| **查询成本** | 低（向量检索） | 中高（图遍历，可能爆炸） |
| **更新成本** | 低（追加 embedding） | 高（关系变更需要重建子图） |

## 四、什么时候用图，什么时候用向量？

### 纯向量就够的场景

- 简单偏好记忆（"我喜欢川菜"）
- 事实陈述（"我是 Java 工程师"）
- 记忆量 < 1000 条
- 不需要关系推理

### 必须上图的场景

- 多跳关系查询（"A 的领导是谁？" → "领导的偏好是什么？"）
- 知识图谱型应用（企业组织架构、项目依赖关系）
- 需要结构化查询（"张三的所有下属"）
- 记忆量 > 1万条，关系复杂

### 混合方案（最佳实践）

```
对话 → 提取
    ├── 向量存储：所有记忆的 embedding（语义检索）
    └── 图存储：实体 + 关系（关系推理）

查询
    ├── 第一阶段：向量检索海选（找语义相关的记忆）
    ├── 第二阶段：图遍历精筛（从海选结果中做关系推理）
    └── 第三阶段：合并返回
```

**mem0 的论文提到的方向**：向量记忆是主存储，Graph Memory 是辅助增强层。不是所有记忆都上图，只把"关系型"记忆上图。

## 五、关键面试追问

**Q18：什么时候该上 Graph Memory，什么时候纯向量就够？**

> 看是否有"关系推理"需求。如果用户只是问"我喜欢什么"，向量检索足够；如果问"我领导的偏好是什么"，需要多跳推理，必须上图。另外看规模：记忆量 <1000 条时，向量+人工维护关系可能够用；>1万条时，自动图提取更可靠。

**Q：Graph Memory 比向量记忆强在哪？举具体例子**

> "A 是 B 的领导，B 负责 X 项目 → X 项目谁拍板？" 向量只能返回"B 负责 X 项目"，但不知道 A 拍板。图可以遍历 A→领导→B→负责→X 项目，得出 A 拍板。

**Q：为什么不是所有记忆系统都上图？**

> 成本。图数据库维护复杂（Neo4j、Dgraph），关系变更需要重建子图。向量库（Chroma、Milvus）简单得多。而且很多记忆系统只需要"偏好记录"，不需要"关系推理"，上图是过度设计。

**Q：混合方案怎么实现？**

> 向量库存所有记忆的语义，图库存实体-关系。查询时先向量检索海选（语义匹配），再对 top-k 结果中的实体做图遍历（关系推理），合并返回。这样既快又准。

## 六、和 K.5 Reranker 的关联

检索链路可以是三段式：

```
向量检索（海选）→ 图遍历（关系精筛）→ Cross-Encoder（语义精排）
```

和 K.5 的 Bi-Encoder → Cross-Encoder 两段式类似，只是中间加了图遍历。
