# Graph Memory（关系图记忆 vs 向量记忆）

> 口述稿 + 对比表
> 日期：2026-06-13
> 关联：向量记忆局限 / 图记忆多跳推理 / 混合方案

---

## 一句话总结

向量记忆擅长语义匹配，图记忆擅长关系推理。多跳查询、结构化查询必须上图，简单偏好记录向量就够。

---

## 口述稿（3 分钟版）

"向量记忆的工作方式是：提取事实 → embedding → 存入向量库。查询时把问题 embedding，做语义相似度检索。这种方式擅长找语义相近的内容，比如'我喜欢游泳'和'我热爱游泳'会被认为是相似的。

但向量记忆有 3 个局限：

**第一，无法区分关系指向**。'张三喜欢游泳'和'游泳喜欢张三'在向量空间里几乎相同，但方向完全相反。向量不保留关系方向。

**第二，无法做多跳推理**。记忆1说'张三是李四的领导'，记忆2说'李四负责 X 项目'。向量检索只能分别返回两条记忆，但无法推理出'张三拍板 X 项目'。因为向量检索是基于语义相似度，不是基于关系遍历。

**第三，无法做结构化查询**。比如'所有在上海上班的人'，向量只能找语义相似，不能像数据库一样做 WHERE city='上海' 的过滤。

**图记忆怎么解决？** 图存储的是节点（实体）和边（关系）。查询时用图遍历语言（Cypher），可以做多跳：张三 -(领导)-> 李四 -(负责)-> X 项目 → 得出张三拍板 X 项目。

**什么时候上图？** 三个标准：① 数据之间存在关联关系（组织架构、项目依赖）② 有结构化查询诉求（'所有下属'）③ 记忆量庞大、关系复杂（人工维护不过来）。小规模简单偏好（'我喜欢川菜'）纯向量就够。

**混合方案**：向量做海选（语义匹配找相关记忆），图做精筛（从相关记忆中做关系推理），Cross-Encoder 做最终排序。三段式检索。"

## 对比表

| 维度 | 向量记忆 | 图记忆 |
|------|----------|--------|
| 存储 | 连续向量空间 | 离散节点+边 |
| 查询 | 语义相似度检索 | 结构化图遍历 |
| 擅长 | 语义匹配、模糊查询 | 关系推理、多跳查询 |
| 不擅长 | 关系方向、多跳推理 | 语义相似度、模糊匹配 |
| 典型场景 | '我喜欢什么' | 'X 项目谁拍板' |
| 存储成本 | 中等 | 高（关系维护） |
| 查询成本 | 低（毫秒级） | 中高（遍历可能爆炸） |

## 面试追问要点

| 追问 | 答案 |
|------|------|
| Q18：什么时候上图？ | ① 关联关系 ② 结构化查询 ③ 记忆量庞大、关系复杂 |
| 向量 vs 图，怎么选？ | 看是否有关系推理需求。简单偏好向量，多跳推理必须图 |
| 为什么不是所有系统都上图？ | 图数据库维护成本高（Neo4j、关系变更重建），小规模向量够用 |
| 混合方案怎么设计？ | 向量海选 → 图遍历精筛 → Cross-Encoder 排序 |
| 图的方向性怎么存？ | 有向边（领导→、负责→），带方向属性 |

---

## 关键数字

- 向量检索延迟：毫秒级
- 图遍历延迟：毫秒~秒级（取决于跳数和图规模）
- 混合方案检索链路：3 段（向量海选 → 图精筛 → 语义精排）
